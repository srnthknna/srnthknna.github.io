fcntcp: TCP's Congestion Control, Rebuilt on UDP

UDP delivers packets with no ordering, no retries, and no guarantee they arrive at all. This project builds reliability on top of it from scratch — its own packet format, checksums, sequence numbers, and a sliding window that behaves like TCP Tahoe: slow start, congestion avoidance, and recovery from both dropped acknowledgments and timeouts.

Because every data packet already carries both endpoints' addresses, ports, sequence number, and the total packet count, the very first packet to arrive doubles as the handshake — no separate connection-setup exchange needed. When the server has every packet, it sends back a plain string, "done", and both sides independently checksum the file to confirm it arrived byte-for-byte.

What "window size" means here

The window is how many packets the sender fires off before it has to stop and wait for an acknowledgment. A bigger window means more data in flight at once — higher throughput, but more to lose if something goes wrong.

Every time an acknowledgment comes back cleanly, the window either doubles (still in "slow start") or grows by one (once past the threshold, in "congestion avoidance"). Every time something looks wrong — three duplicate acks, or no ack at all before the timeout — the window is slashed back down to 1 and the threshold is halved, so the next climb is more cautious.

A real transfer, replayed

This is the actual congestion window from a real run of this program — a 2.96 MB file, sent as 5,928 packets over UDP. Each point is one acknowledgment cycle; every value below comes straight from that transfer's console log, not a simulation.

Dashed line marks the slow-start threshold (128 packets). The window doubles each cycle up to that point, then grows by only 1 per cycle afterward — until a timeout on the very last outstanding packet drops it straight back to 1.

Client and server independently computed the same result: 2,963,706 bytes, CRC32 ca71963d — on both ends, from a file sent entirely over "unreliable" UDP.

Java implementation: UDP DatagramSockets with Java object serialization for packets, CRC32 checksums, and separate sender / receiver / timer threads coordinating through shared, synchronized state.
