Eliminating Partial Redundancy in MLton

MLton is a whole-program optimizing compiler for Standard ML. This project adds GVN-PRE, an optimization pass that catches redundant computations ordinary common-subexpression elimination has to leave behind — because they're only redundant along some paths, not all of them.

Full vs. partial redundancy

An expression is fully redundant if it's already been computed on every path leading to it — safe for any compiler to remove. It's partially redundant if that's only true on some paths. Plain CSE can only clean up the first kind.

Full redundancy

if (cond)
X = A+B
Z = A+B
Y = A+B

Computed on both branches — CSE removes it immediately.

Partial redundancy

if (cond)
X = A+B
nothing
Y = A+B

Only one branch computes it — CSE can't touch this. GVN-PRE can.

The four-phase pass

Once a redundancy is found, MLton's SSA form runs through four phases to remove it — matching what just happened in the diagram above.

BuildSets I
Walk the dominator tree, record what each block computes and what's already available.
→
BuildSets II
Work out which expressions could be hoisted into each block, iterating to convergence.
→
Insert
Add the missing computation to whichever branch doesn't already have it.
→
Eliminate
Replace every redundant statement with a reference to the value that already exists.

Does it actually help?

On a synthetic benchmark built to contain real partial redundancy, GVN-PRE cut runtime to roughly a third of the unoptimized baseline — about 3x faster, and clearly ahead of CSE alone.

Across MLton's full 43-program benchmark suite the picture was more mixed — about half the programs got faster, since inserting code on missing branches can add register pressure that eats into the win.
